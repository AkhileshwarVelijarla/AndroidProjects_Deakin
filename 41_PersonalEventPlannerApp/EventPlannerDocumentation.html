Personal Event Planner App Documentation

1. Project Overview

This Android application is a Personal Event Planner App built with MVVM architecture, DataBinding, Room Database, Jetpack Navigation, Fragments, and Material components. It allows users to create, view, update, and delete upcoming events while persisting all data locally on the device.

2. Architecture Summary

The project is divided into three major MVVM layers:

  • View: MainActivity, Fragments, XML layouts, RecyclerView adapter
  • ViewModel: EventViewModel manages UI state, validation, and actions
  • Model: Room classes including entity, DAO, database, and repository

3. Full Data Flow

Create Event Flow

  1. User opens the Add Event screen from bottom navigation.
  2. User enters Title, Category, Location, and picks Date/Time.
  3. DataBinding updates the matching fields in EventViewModel.
  4. User taps Save.
  5. EventViewModel.saveEvent() validates title and date rules.
  6. If valid, the ViewModel creates an EventEntity.
  7. The ViewModel calls EventRepository.insert().
  8. The repository calls EventDao.insert() on a background thread.
  9. Room writes the event to the local SQLite database.
  10. LiveData<List<EventEntity>> emits the updated list.
  11. EventListFragment observes the new list and refreshes RecyclerView automatically.

Read Event Flow

  1. EventDao.getAllEvents() returns a LiveData list sorted by date ascending.
  2. The repository exposes this list to the ViewModel.
  3. The ViewModel exposes it to the fragment.
  4. EventListFragment observes the data.
  5. Whenever the database changes, RecyclerView receives the new list and redraws.

Update Event Flow

  1. User taps Edit on an event card.
  2. EventListFragment navigates to the Add/Edit screen with the event ID.
  3. AddEditEventFragment reads the ID from fragment arguments.
  4. EventViewModel.prepareForm(eventId) loads the event from Room.
  5. The ViewModel pushes values into DataBinding fields.
  6. User modifies details and taps Save.
  7. saveEvent() detects edit mode and calls repository.update().
  8. Room updates the row and the list UI refreshes through LiveData.

Delete Event Flow

  1. User taps Delete on an event card.
  2. EventListFragment calls viewModel.deleteEvent(event).
  3. The repository calls EventDao.delete() on the database executor.
  4. Room removes the row.
  5. LiveData emits the reduced list.
  6. The UI updates automatically and a Snackbar confirms deletion.

4. How Navigation Works

The app uses a single Activity and multiple Fragments. MainActivity hosts a NavHostFragment and a BottomNavigationView. The navigation graph defines two destinations:

  • eventListFragment for the dashboard
  • addEventFragment for add and edit operations

Because the bottom navigation menu IDs match the fragment destination IDs, the call to NavigationUI.setupWithNavController() automatically connects the tabs to navigation.

5. How DataBinding Works

Layouts wrapped with a root <layout> tag generate binding classes such as ActivityMainBinding, FragmentEventListBinding, FragmentAddEventBinding, and ItemEventBinding.

  1. Each layout declares a viewModel variable where required.
  2. The fragment sets binding.setViewModel(viewModel).
  3. UI fields read from and write to MutableLiveData in the ViewModel.
  4. Two-way binding keeps form fields synchronized with ViewModel state.

6. How Room Persistence Works

Room is the local persistence layer. The app stores events in a database table called events. Since Room writes to SQLite on the device, the data remains available after the app is closed or the device is restarted.

  • EventEntity defines the table structure
  • EventDao defines the database operations
  • EventDatabase creates the Room database singleton
  • EventRepository abstracts DAO access from the ViewModel

7. Validation and Error Handling

The app applies validation in two layers:

  • UI prevention: the date picker blocks past dates with setMinDate()
  • Logic validation: EventViewModel.saveEvent() checks:
    • title is not empty
    • date/time is selected
    • selected date/time is not in the past

Snackbar messages are used for validation errors, successful saves, updates, and deletions.

8. File-by-File Explanation

Build and Configuration Files

app/build.gradle.kts: enables DataBinding and adds dependencies for Lifecycle, Navigation, RecyclerView, Material, and Room.

gradle/libs.versions.toml: central dependency version catalog.

build.gradle.kts: project-level Gradle configuration.

settings.gradle.kts: repository setup and module inclusion.

Manifest and Entry Point

AndroidManifest.xml: declares the app, theme, and launcher activity.

MainActivity.java: hosts navigation and connects bottom navigation with the nav controller.

activity_main.xml: contains the FragmentContainerView and BottomNavigationView.

Navigation Resources

nav_graph.xml: defines the two fragments and the optional eventId argument.

bottom_nav_menu.xml: bottom bar items for Event List and Add Event.

Room Data Layer

EventEntity.java: Room entity for one event row with title, category, location, and event date/time.

EventDao.java: SQL access methods for insert, update, delete, query all, and query by ID.

EventDatabase.java: singleton Room database and background executor.

EventRepository.java: data access abstraction used by the ViewModel.

ViewModel

EventViewModel.java: main business logic file. It stores form state, event list state, edit mode state, validation logic, save/update/delete actions, empty-state flags, and one-time UI events.

UI Layer

EventListFragment.java: shows the dashboard, observes LiveData, handles edit/delete actions, and shows messages.

AddEditEventFragment.java: handles add/edit form behavior, date and time pickers, save flow, clear flow, and post-save navigation.

EventAdapter.java: RecyclerView adapter based on ListAdapter with DiffUtil.

item_event.xml: layout for each event card.

fragment_event_list.xml: dashboard screen layout using DataBinding.

fragment_add_event.xml: add/edit form layout using DataBinding and two-way binding.

Utility Classes

DateTimeFormatterUtil.java: converts epoch milliseconds into user-friendly date/time text.

EventValidator.java: reusable validation helper methods.

OneTimeEvent.java: avoids repeating one-shot Snackbar events after re-observation.

Resources

strings.xml: user-facing text.

colors.xml: app palette.

bottom_nav_colors.xml: color selector for bottom nav states.

themes.xml and values-night/themes.xml: app theme configuration.

category_chip_background.xml: rounded chip drawable for the category label.

Testing

EventValidatorTest.java: unit tests for title and date validation logic.

9. Why the Assignment Requirements Are Satisfied

  • CRUD is implemented through the list screen and add/edit screen.
  • Room is used for local persistence.
  • Jetpack Navigation with Fragments and Bottom Navigation is implemented.
  • Validation blocks empty title, missing date, and past dates.
  • Snackbar feedback is used for save, update, delete, and validation errors.
  • DataBinding and MVVM architecture are used throughout the app.

10. Short Viva Explanation

The app is built with single-activity MVVM architecture. MainActivity hosts a navigation container and bottom navigation. EventListFragment displays all stored events from Room through LiveData. AddEditEventFragment handles both creating and updating events. The EventViewModel stores form state, applies validation, and decides whether to insert or update records. The repository communicates with Room, and the UI updates automatically whenever the database changes.
